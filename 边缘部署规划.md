# 边缘部署规划与轻量化路线（XH-202615）

> 创建：2026-06-28。缘起：用户提出「项目终态要边缘部署，技术路线是否包含？」
> 性质：战略级记录 + 待确认项 + W8 模块建议。在目标硬件确认前为「规划草案」，确认后定稿。

---

## 1. 核心结论（先看这条）

当前技术路线（00 总纲 W1–W7）**有轻量化的零散种子，但缺系统性的边缘部署规划**。要正式纳入，取决于「边缘」的确切定义（家电 MCU？网关盒子？本地服务器？），需向主办方/团队确认目标硬件后才能定稿。

---

## 2. 现状评估

### ✅ 已有的轻量化种子（非系统性规划）
1. **效率是评分项（20%）**：CLAUDE.md 差异化策略 D5 = 效率优化（TS-RNNT 形态 Hadamard 积预注册 + Whisper 量化/蒸馏/流式）。但目标是**云端 L20 提速**，不是边缘设备。
2. **本机实测 DiCoW_v3_2（turbo）已是轻量级**：RTF=0.058 / params **0.89G** / GPU 峰值 **2.13GB**。体积和速度已接近边缘可部署量级，但当前路线没把它往边缘方向显式规划。
3. **wespeaker 声纹天生边缘友好**：6.6M 参数 / 26MB。Part1 的 enrollment→target 锁定环节本身是边缘友好的（轻量声纹 + 余弦匹配）。
4. **DiariZen diarization**：WavLM-large backbone 偏重（~90M+），是边缘化的主要瓶颈之一。

### ❌ 当前技术路线缺失的边缘部署内容
- **目标硬件未定义**：边缘指什么？① 家电主控 MCU（算力极弱，<1G RAM）② 边缘网关盒子（中端 SoC，1–4G RAM）③ 本地服务器/高端边缘（强 SoC，8G+）。三者算力差几个数量级，方案完全不同。题目和现有文档都没界定。
- **量化/蒸馏/ONNX/INT8 没排进 W1–W7 行动地图**：D5 只是「可选」，无具体里程碑、无实测计划。
- **流式/低延迟未规划**：边缘部署通常要求流式 + 低延迟唤醒；当前 `enroll_infer.py` 是**离线整段处理**（载入整条音频→diarization→转写）。
- **模型体积/内存预算无约束**：没有「目标 < X MB / < Y MB 内存 / < Z W 功耗」这类边缘硬约束设计目标。

---

## 3. 核心张力：参赛拿分 vs 边缘部署

| 维度 | 参赛评测（现状） | 边缘部署（终态） |
|---|---|---|
| 硬件 | L20 GPU（云端数据中心卡，80GB 显存级） | 待定（MCU/网关/服务器，算力差几个数量级） |
| 优化方向 | 精度（CER 40% + 拒识 40%） | 轻量、低延迟、低内存、低功耗 |
| 处理方式 | 离线整段 | 流式 + 唤醒 |
| 冲突 | 精度版（大模型、强算力） | 轻量版（小模型、弱算力） |

**结论**：参赛拿分和最终边缘部署的优化方向有冲突。需在方案里显式做一条 **「精度版（参赛）→ 轻量版（部署）」** 的演进路线，否则答辩时评委可能问「这套 pipeline 怎么落到家电上」而答不上。

---

## 4. 待确认（阻塞定稿）⚠️

**目标硬件定义**——向主办方/团队确认：
- 家电 MCU（算力极弱）？→ 需极端轻量（KWS 前端 + 云端 ASR，或极小模型）
- 边缘网关盒子（中端 SoC）？→ INT8 量化 + 蒸馏小模型可本地跑
- 本地服务器/高端边缘？→ 当前 turbo 版近乎可直接用

三者方案完全不同，确认前边缘部署路线无法定稿。

## 5. W8 部署轻量化模块（建议纳入技术路线）

建议在 00 总纲新增 **W8 部署轻量化**，覆盖端到端落地：

| 子模块 | 内容 | 产出 |
|---|---|---|
| W8.1 量化 | Whisper + DiariZen + wespeaker 的 INT8/FP16 量化 | 体积/速度/CER 权衡表 |
| W8.2 蒸馏 | 小模型替代（Whisper-tiny/base 替 turbo？或 TS-RNNT 形态） | 蒸馏小模型 + CER 对比 |
| W8.3 导出 | ONNX / TensorRT / NCNN（依目标硬件） | 可部署推理引擎 |
| W8.4 流式改造 | chunked streaming + 低延迟唤醒（KWS 前端衔接） | 流式 pipeline + 延迟测量 |
| W8.5 目标硬件基准 | 目标硬件上的 RTF/延迟/内存/功耗实测 | 部署基准报告 |

**双重价值**：W8 同时是 **D5 效率优化（20% 评分）的实质化** + **答辩差异化亮点**（不只是「跑得快」，而是「能落家电」）。

## 6. 当前可立即做（不依赖硬件确认）

这些是「无论如何都有价值」的轻量化铺垫，可在 W1–W7 推进中顺手做：
1. **记录本机轻量化基线**（已有）：DiCoW turbo 0.89G / RTF 0.058 / GPU 峰值 2.13GB
2. **端到端 RTF 测量**：`enroll_infer.py` 全链路（diarization + 声纹匹配 + DiCoW）的 RTF/延迟，区分「可流化部分」与「必须整段部分」
3. **量化可行性试探**：Whisper INT8 量化对中文 CER 的影响（小规模实验，量化是否伤精度）
4. **环节轻量度盘点**：哪些环节已边缘友好（wespeaker）、哪些是瓶颈（WavLM diarization、Whisper-large）

---

## 7. 答辩预案

- **评委问「这套 pipeline 怎么落家电」**：呈现「精度版（L20 参赛）→ 轻量版（INT8 + 蒸馏 + 流式）」演进路线 + 已有轻量基线（turbo 0.89G / RTF 0.058）+ wespeaker/enrollment 天生轻量 + Whisper 有成熟量化蒸馏路径。
- **诚实边界**：100% 完全重叠单通道死区（T15 已论证）→ 边缘端需多麦克风/空间前端（DSENet，待通道数确认）。
- **差异化**：不只「跑得快」，而是给出「精度-轻量」可权衡的演进路径，体现工程落地思考。
